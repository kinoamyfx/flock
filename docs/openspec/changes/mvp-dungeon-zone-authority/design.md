## Context
目标玩法：单人经营（可招募门派成员）、2D 俯视（酷炫灯光/迷雾）、秘境合作为主但允许 PvP 杀人夺宝，死亡“当前携带全部物品”掉落。

强约束：
- 最大限度防作弊。
- 断线后角色留在原地 60 秒，可被击杀并触发掉落（安全箱除外）；重连按“未重置原地恢复/已重置入口恢复”。
- “秘境地图不限制人数”的产品口径，工程上通过“自动分线”实现；MVP 单线软上限 64 人。
- 自研为主，引入 Godot 作为渲染层；PC 优先。
- 传输层需可替换：优先可靠 UDP；MVP 可先用纯 C# 实现跑通语义，后续替换为 ENet/QUIC 不影响协议与业务逻辑。

## Goals / Non-Goals
- Goals:
  - 在“全掉落”的高风险规则下，建立最小可靠的权威判定与结算能力，避免复制/伪造/回滚导致经济崩溃。
  - 秘境支持多人同图体验，并能通过分线扩容，避免单机房主/纯P2P带来的作弊与稳定性问题。
  - 将玩法规则固化为可验证的规范（OpenSpec），便于后续实现与回归。
- Non-Goals:
  - 本变更不定义完整的数值/掉落表/怪物库，仅定义系统边界与反作弊约束。
  - 本变更不要求一次性实现所有视觉特效，仅定义“渲染层职责边界”与必要的数据接口。

## Decisions
- Decision: 采用“服务器权威（Authoritative Server）”的秘境战斗与结算模型。
  - Why: 全掉落把作弊收益放大到不可接受；必须将掉落、背包、死亡判定放在服务端可信域。
- Decision: “同一张秘境地图”的产品表达通过“自动分线”实现；单线软上限 64 人。
  - Why: 动作战斗+多人场景的带宽与判定复杂度极高；分线是最小工程可控方案。
- Decision: 断线后角色留身在场，可被击杀并掉落。
  - Why: 这是明确的反逃逸规则，降低拔线规避风险的收益。
- Decision: 自研 Headless 模拟内核，Godot 仅做渲染/输入/表现。
  - Why: 降低引擎锁定与成本，保证核心模拟在服务端与离线仿真一致；渲染可快速迭代。
- Decision: 传输层选择“可靠 UDP”，并保持可替换；MVP 使用 LiteNetLib（纯 C#）跑通连接/断线/握手/重连。
  - Why: 当前 macOS arm64 环境下 ENet 的可用 C# 绑定存在平台/架构/协议兼容性问题；先保证语义闭环与可测试性，后续再替换为 ENet/QUIC。

## Alternatives considered
- Alternative: 纯 P2P 或房主权威 + 服务器仅结算
  - Rejected: 作弊面过大（改伤害/改掉落/改拾取/回滚），且全掉落会导致不可接受的公平性风险。
- Alternative: 不限制人数且单实例不分线
  - Rejected: 单线并发会导致 AOI/技能广播/碰撞与判定成本爆炸，MVP 难以稳定交付。

## Risks / Trade-offs
- 高并发动作战斗的延迟与手感：
  - Mitigation: 服务端权威 + 客户端预测（仅表现层）+ 回滚校正；AOI 裁剪与技能广播降噪。
- “全掉落”导致新手劝退与极端保守玩法：
  - Mitigation: 设计安全区/保险格/托管仓等机制作为经济阀门（本变更只约束系统能力，具体数值另立变更）。
- 服务器成本与运维复杂度上升：
  - Mitigation: 分线与弹性扩缩容；先以单机多进程实现 MVP，再演进到多机。

## Migration Plan
1. 固化规范（本变更）：定义 world tick、zone server、pvp-loot、economy、client-render 的 requirements。
2. MVP 实现：单机可跑的 Zone Server + 客户端连接 + 64人软上限分线 + 断线留身 + 掉落结算落库。
3. 扩展：多机分布式分线、监控告警、压测与作弊对抗迭代。

## Open Questions
（2026-01-02 已决策并锁定）
- 安全箱：介子袋，9 格；材料制作；允许在秘境内存取与整理。
- 撤离/退出：允许撤离点（传送阵）；长读条 + 可被打断 + 高成本。
- 死亡掉落拾取保护：死亡后 10 秒内仅“击杀者队伍”可拾取。
- 秘境重置：无人后 10 分钟，或通关后 10 分钟。
- 经济：交易收税 + 手续费；税与手续费进入系统销毁（资源回收池）。
- Mod：MVP 先表现层；后续再扩展逻辑层；Mod 本身不允许联网。
